    pour personnaliser sans tout reecrire.

    Mettez a jour la logique metier sans recompiler.

SECTION 3 - ARCHITECTURE DU PIPELINE (schema interactif)
================================================================================

[LAYOUT]
Section full-width avec un titre centre, puis un schema interactif du pipeline
occupant toute la largeur sous le titre. Fond #0D1117.

[VISUEL]
Pipeline horizontal interactif (librairie : D3.js ou Cytoscape.js).
Les 8 etapes sont des noeuds relies par des fleches :

    protocol -> data_format -> caller -> operation -> business -> security -> multiplex -> execution

Les noeuds sont colores par responsabilite :
- Bleu clair   : Transport (protocol, data_format)
- Vert         : Contexte (caller, operation)
- Jaune        : Metier (business)
- Rouge        : Securite (security)
- Orange       : Routage (multiplex)
- Cyan         : Execution (execution)

Le noeud "execution" se divise en 4 branches :
    execution -> CRUD (create, readAll, readOne, update, deleteOne, deleteAll)
    execution -> Use Cases
    execution -> Workflows
    execution -> Authentication

Interactions :
- Hover sur un noeud : le noeud s'agrandit, un tooltip affiche sa description
  et les scripts associes.
- Click sur un noeud : ouvre un panneau lateral avec la liste des fonctions
  disponibles dans cette etape.
- Mode A / Mode B : deux boutons au-dessus permettent de visualiser
  le pipeline complet (Mode A : requete brute) ou le pipeline reduit
  (Mode B : requete pre-traitee, saute les 3 premieres etapes).

[TEXTE]

    Un pipeline en 8 etapes

    Chaque requete traverse un pipeline deterministe. Les etapes 1 a 3
    gerent le transport et le contexte. Les etapes 4 a 6 appliquent
    les regles metier et la securite. Les etapes 7 et 8 routent vers
    le bon handler et executent la logique.

    Chaque etape est un script. Modifiable. Testable. Independant.

[LEGENDE sous le schema]

    protocol            Extraction des donnees brutes (headers, body, params)
    data_format         Deserialisation vers DTO / objets domaine
    caller              Construction du contexte appelant (tenantId, ownerId, autorites)
    operation           Detection de l'operation CRUD / use case / workflow
    business            Validation tenant / owner / contraintes metier
    security            Authentification et autorisation
    multiplex           Routage vers le bon handler d'execution
    execution           Scripts CRUD, use cases, workflows, ou authentification

SECTION 4 - MULTI-TENANCY EN DETAIL (differenciateur cle)
================================================================================

[LAYOUT]
Section split : texte a gauche (55%), visuel a droite (45%).
Fond #111827.

[VISUEL - cote droit]
Tableau / matrice interactive montrant les combinaisons de visibilite :

    +------------------+--------+---------+-------+-----------+
    | Configuration    | Public | Tenant  | Owned | Hiddenable|
    +------------------+--------+---------+-------+-----------+
    | Blog public      |   X    |         |       |     X     |
    | Donnees tenant   |        |    X    |       |           |
    | Documents perso  |        |    X    |   X   |           |
    | Posts partages   |        |    X    |       |     X     |
    +------------------+--------+---------+-------+-----------+

Sous le tableau, schema du filtre genere automatiquement pour chaque
combinaison. Au hover sur une ligne, le filtre correspondant s'affiche.

[TEXTE - cote gauche]

    Isolation des donnees, integree au framework

    Chaque entite declare sa strategie de visibilite via des
    annotations. Le framework genere automatiquement les filtres
    d'acces appliques a chaque requete.

    5 annotations combinees librement :

    @EntityPublic       Visible par tous, sans filtre tenant
    @EntityTenant       Appartient a un tenant (filtre tenantId)
    @EntityOwned        Appartient a un utilisateur (filtre ownerId)
    @EntityShared       Partageable entre tenants (champ shareWith)
    @EntityHiddenable   Masquable (champ hidden, visible au proprietaire)

    Deux roles privilegies :

    - Super-tenant : accede a toutes les donnees, tous les tenants
    - Super-owner : accede a toutes les entites, tous les proprietaires

    Le filtre est applique avant l'execution. Pas de risque d'oubli.
    Pas de fuite de donnees entre tenants.

    Pas besoin de multi-tenancy ?

    Desactivez-le en une ligne :

        ApiContextBuilder.builder()
            .multiTenant(false)    // desactive tout le tenant
            ...

    Le framework passe en mode mono-tenant : plus de filtres tenant,
    plus de super-tenant, plus de shared. Les filtres owner et
    visibilite restent actifs. Mode strict : toute tentative
    d'appeler superTenantId() ou tenant(true) leve une exception.

SECTION 5 - DSL BUILDER (experience developpeur)
================================================================================

[LAYOUT]
Fond #0D1117. Bloc central avec un snippet de code principal (70% de la largeur)
et des annotations explicatives en marge.

[VISUEL]
Terminal-style code block (fond #161B22, bordure arrondie, barre de titre
avec 3 points rouge/jaune/vert). Coloration syntaxique Java.
Des lignes de rappel (thin lines) partent du code vers des annotations
explicatives dans la marge droite.

[TEXTE]

    Tout se configure via un DSL fluide

    Un seul point d'entree. Type-safe. Navigable avec up().
    Pas de fichiers XML, pas d'annotations dispersees,
    pas de magie de classpath scanning.

[CODE]

    IApiContext context = ApiContextBuilder.builder()
        .superTenantId("SUPER_TENANT")               // <- Identifiant du tenant admin
        .domain(User.class)                           // <- Domaine auto-nomme "users"
            .entity()
                .id("id")                             // <- Cle primaire
                .uuid("uuid")                         // <- Identifiant metier (auto-genere)
                .tenantId("tenantId")                 // <- Champ d'isolation tenant
                .ownerId("ownerId")                   // <- Champ de propriete
            .up()
            .dto(UserDto.class)                       // <- DTO avec mapping automatique
                .id("id")
                .uuid("uuid")
                .db(new MongoDao())                   // <- Connecteur de persistance
            .up()
            .security()                               // <- Securite du domaine
                .authentication()
                    .loginPassword()                  // <- Methode d'authentification
                .up()
                .authorization()
                    .jwt()                            // <- Autorisation JWT
                    .signable()
                    .refreshable()
                .up()
            .up()
            .creation(true)                           // <- Active le CRUD create
            .readAll(true)                            // <- Active le CRUD readAll
            .readOne(true)                            // <- Active le CRUD readOne
            .workflow("deleteAll")
                .security().disable(true).up()        // <- Desactive la securite
            .up()
        .up()
        .build();                                     // <- Build immutable

    context.onInit();
    context.onStart();

[CODE - DEUXIEME BLOC]

    Requetes fluides : du builder a l'execution

    Construisez et executez vos requetes en une seule chaine fluide.
    Les raccourcis CRUD evitent la construction manuelle des
    OperationDefinition. Disponible sur IDomainContext et IApiContext.

    // Depuis un domaine
    products.request()
        .createOne(myProduct)                   // <- Operation + body en un appel
        .caller(caller)                         // <- Contexte appelant
        .execute();                             // <- Build + execute

    // Lecture avec filtres
    products.request()
        .readAll()                              // <- Operation readAll
        .filter(myFilter)                       // <- Filtre optionnel
        .page(pageable)                         // <- Pagination
        .sort(sort)                             // <- Tri
        .caller(caller)
        .execute();

    // Raccourci depuis le contexte API
    context.request("products")                 // <- Acces direct par nom de domaine
        .deleteOne("uuid-123")
        .caller(caller)
        .execute();

    // Build en deux temps (inspection avant execution)
    IRequest request = products.request()
        .updateOne("uuid-123", body)
        .caller(caller)
        .build();                               // <- IRequest inspectable

    IOperationResponse response = request.execute();

[TEXTE + CODE]

--- Carte 1 : Login / Password ---

    [icone: cadenas]

    Login / Password

    Authentification classique par identifiant et mot de passe.
    Encodage configurable (Bcrypt, SHA, PBKDF2). Champ "enabled"
    pour desactiver un compte sans le supprimer.

    .authentication()
        .loginPassword()
        .authenticator(User.class)
    .up()

--- Carte 2 : PIN ---

    [icone: grille de chiffres]

    Code PIN

    Authentification par code numerique avec compteur d'erreurs.
    Blocage automatique apres N tentatives echouees.
    Ideal pour les applications mobiles ou les terminaux.

    .authentication()
        .pin()
        .authenticator(User.class)
    .up()

--- Carte 3 : Challenge-Response ---

    [icone: fleches bidirectionnelles]

    Challenge-Response

    Le serveur genere un defi avec une duree de validite.
    Le client repond avec la solution. Adapte aux scenarios
    MFA ou aux integrations avec des systemes externes.

    .authentication()
        .challenge()
        .authenticator(User.class)
    .up()

--- Carte 4 : JWT Authorization ---

    [icone: jeton / badge]

    JWT Authorization

    Signature et validation de tokens JWT. Algorithmes RSA
    ou HMAC. Refresh tokens pour le renouvellement sans
    re-authentification. Claims customisables.

    .authorization()
        .jwt()
        .signable()
        .refreshable()
    .up()

SECTION 7 - MODULES (architecture modulaire)
================================================================================

[LAYOUT]
Fond #0D1117. Schema en blocs empiles montrant la structure modulaire.
Chaque bloc est cliquable pour afficher plus de details.

[VISUEL]
Architecture en couches, de bas en haut :

    +-------------------------------------------------------+
    |              garganttua-api-native-image               |  <- GraalVM
    +-------------------------------------------------------+
    |   garganttua-api-spring  (REST, Swagger, MongoDB)     |  <- Integration
    +-------------------------------------------------------+
    |   garganttua-api-security (login, PIN, JWT, etc.)     |  <- Securite
    +-------------------------------------------------------+
    |              garganttua-api-core                       |  <- Moteur
    |   (builders, contexts, pipeline, scripts, repository) |
    +-------------------------------------------------------+
    |              garganttua-api-spec                       |  <- Contrats
    |   (interfaces, annotations, enums, definitions)       |
    +-------------------------------------------------------+
    |              garganttua-core                           |  <- Fondation
    |   (reflection, injection, script, workflow, mapper)    |
    +-------------------------------------------------------+

Hover sur un bloc : bordure en couleur d'accent, description etendue.
Click : ouvre un panneau avec la liste des classes/interfaces principales.

[TEXTE]

    Architecture en couches strictes

    Chaque couche depend uniquement des couches inferieures.
    Le module spec est un pur contrat : zero logique, zero dependance
    externe. Le core implemente le moteur. Les modules Spring et
    securite sont optionnels et interchangeables.

[DESCRIPTION PAR MODULE]

    garganttua-api-spec      Contrats purs : 40+ annotations, 60+ interfaces,
                             definitions immutables. Aucune logique metier.

    garganttua-api-core      Moteur du framework : builders DSL, contexts runtime,
                             pipeline scriptable, repository, expressions.

    garganttua-api-security  Authentification pluggable (login/password, PIN,
                             challenge) et autorisation JWT. 7 sous-modules.

    garganttua-api-spring    Integration Spring Boot 3.3 : REST controllers
                             auto-generes, Swagger/OpenAPI, MongoDB DAO,
                             Spring Security adapter.

    garganttua-api-native    Configuration GraalVM Native Image pour un
                             demarrage en millisecondes.

--- Etape 1 ---

    [icone: fusee]

--- Etape 4 ---
    [icone: flux de donnees]

    4. Servez

    Chaque requete traverse le pipeline en 8 etapes.
    Filtrage tenant automatique. Securite appliquee.
    CRUD, use cases, ou workflows executes.
